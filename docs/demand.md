## 一、产品概述

### 1.1 产品定义

Codex Island 是一款 macOS 原生菜单栏应用，驻留在 MacBook Pro 刘海区域（Notch），实时监听并可视化 Codex 桌面应用的工作状态。产品以**像素宠物**为核心视觉符号，将 Codex 的 AI 工作状态（思考中、输出中、等待用户）映射为宠物的情绪与动作，同时实时展示 Token 消耗数据，并在 Codex 需要用户介入时第一时间提醒。

### 1.2 产品口号

> "你专注写代码，宠物帮你盯着 AI。"

### 1.3 目标用户

**主要用户**：每日使用 Codex 桌面应用进行编程辅助的开发者，尤其是：

- 使用 Codex 处理复杂任务（需要多轮工具调用审批）时容易分心的用户
- 关注 API Token 成本和使用效率的用户
- 使用 MacBook Pro（2021年及之后，带刘海）的用户

**次要用户**：使用无刘海 Mac 的 Codex 用户（通过浮动胶囊形态支持）

### 1.4 核心价值主张

| 痛点 | Codex Island 的解决方案 |
|------|------------------------|
| Codex 等待用户审批时，用户不知道（在别的窗口工作） | 灵动岛脉冲红色提示 + 系统通知，第一时间感知 |
| 不知道当前会话消耗了多少 Token | 实时数字显示，精确到输入/缓存/输出三个维度 |
| 切换到 Codex 窗口才能知道 AI 的状态 | 刘海区域常驻，一眼看出宠物状态即 AI 状态 |
| 不清楚 Prompt Cache 命中情况 | 专门展示缓存命中率，帮助优化对话策略 |

## 三、功能需求详述

### 3.1 数据采集

#### 3.1.1 JSONL 文件监听（主路径）

**需求描述**：App 启动后自动扫描 `~/.codex/sessions/` 目录下近期会话 JSONL 文件，持续监听新增行，解析 `token_count` 事件。阶段一 UI 只显示一个胶囊，但数据层必须按 `session_id` 支持多个 Codex 会话同时进行。

**功能点**：
- 自动定位近期活跃的 `rollout-*.jsonl` 文件，并在启动时选取最新活跃会话作为初始显示对象
- 增量读取（从文件末尾开始，不重读历史）
- 新 `.jsonl` 文件创建时（新会话开始）自动纳入监听
- 当 JSONL 尚未写入会话元数据时，可从 `rollout-...-UUID.jsonl` 文件名中兜底提取 `session_id`
- 解析 `token_count` 事件，兼容 Codex 实际的 `payload.info.total_token_usage` 嵌套结构，并归一化为四个字段：`input_tokens` / `cached_input_tokens` / `output_tokens` / `reasoning_tokens`
- 启动时回放最新脱敏 Token 检查点，避免统计在下一次输出前全为 0
- 检测 `task_started` / `task_complete` / `awaiting_approval` 事件，驱动运行、完成与等待状态
- 多会话同时活跃时，胶囊显示优先级最高且最近活动的会话；多个胶囊或会话列表不属于阶段一

**数据精度要求**：
- 累计 Token 数与 Codex 界面显示误差 < 5%
- 单轮增量值（差分）始终为非负数

**容错要求**：
- JSONL 中存在非法 JSON 行时，跳过该行不崩溃
- 文件被删除/轮转时，自动寻找新文件
- `~/.codex/sessions/` 目录不存在时，显示友好提示，不崩溃

#### 3.1.2 App-Server WebSocket（辅助路径）

**需求描述**：尝试连接 Codex App-Server 的 WebSocket 接口（`ws://127.0.0.1:4500`），获取更实时的 Turn 状态变化通知。

**功能点**：
- 自动尝试连接，失败时静默降级（仅使用 JSONL 路径）
- 断连后自动重连（间隔 3 秒）
- 接收 `turn/start`、`turn/stop`、`tool/approval/request` 等事件
- WebSocket 事件优先级高于 JSONL 推断（更实时）

**降级策略**：WebSocket 不可用时，系统仍完整工作，仅靠 JSONL 路径，延迟约 100-200ms。

#### 3.1.3 Streaming 超时兜底

**需求描述**：防止状态机卡在 Streaming 状态。

**规则**：若连续 4 秒内 `output_tokens` 无增加，自动将状态切换为 Idle。

### 3.2 会话状态机

#### 3.2.1 五个状态定义

| 状态 ID | 状态名 | 触发条件 | 退出条件 |
|---------|--------|----------|----------|
| `idle` | 空闲 | 初始状态；AI 回复完成；用户无新提问 | 检测到 `user_message` 事件 |
| `thinking` | 思考中 | 检测到 `user_message` 或 `turn/start` | 检测到 output token 增加 |
| `streaming` | 输出中 | output token 开始增加 | AI 回复停止；超时 4 秒无增量 |
| `awaiting_input` | 等待用户 | 检测到 `awaiting_approval` 或 `tool/approval/request` | 用户审批；用户取消 |
| `error` | 错误 | 连接断开超时；JSONL 解析连续失败 | 3 秒后自动回 Idle |

#### 3.2.2 状态转换规则

- 任何状态均可直接切换到 `awaiting_input`（优先级最高）
- `awaiting_input` → `thinking`（用户审批后）
- `awaiting_input` → `idle`（用户取消后）
- `error` 状态 3 秒后自动转为 `idle`
- 相同状态不重复触发 UI 切换

### 3.3 Token 数据展示

#### 3.3.1 展示字段

| 字段名 | 说明 | 展示位置 |
|--------|------|----------|
| 总输入 Tokens | `input_tokens` 累计值 | Pill 主区 / Expanded 卡片 |
| 缓存命中 Tokens | `cached_input_tokens` 累计值 | Pill 主区 / Expanded 卡片 |
| 未缓存 Tokens | `input_tokens - cached_input_tokens` | Expanded 卡片 |
| 输出 Tokens | `output_tokens` 累计值 | Pill 主区 / Expanded 卡片 |
| 缓存命中率 | `cached / input × 100%` | Pill 缓存字段后缀 / Expanded |
| 总计 Tokens | `input + output` 累计 | Pill 右侧 / Expanded |

**说明**：所有展示值均为当前会话累计值（非单轮增量），以便用户评估总成本。

#### 3.3.2 数字格式化规则

| 范围 | 格式 | 示例 |
|------|------|------|
| 0 – 999 | 原始整数 | `342` |
| 1,000 – 9,999 | 一位小数 + K | `1.2K` |
| 10,000 – 999,999 | 整数 + K | `12K` |
| ≥ 1,000,000 | 一位小数 + M | `1.2M` |

#### 3.3.3 数字动画要求

- 数值变化时有**向上滚动**的数字动画（`contentTransition(.numericText)`）
- 数值变化瞬间有 **0.3 秒白色高亮闪烁**，随后恢复对应颜色
- 动画时长：spring(response: 0.4, dampingFraction: 0.8)，不可过快（防眩晕）

#### 3.3.4 颜色规范

| 字段 | 颜色 | 含义 |
|------|------|------|
| 输入 Tokens | `#60A5FA`（蓝） | 中性，信息输入 |
| 缓存命中 | `#34D399`（绿） | 积极，省钱 |
| 未缓存 | `#FBBF24`（黄） | 注意，实际消耗 |
| 输出 Tokens | `#FB923C`（橙） | 产出 |
| 总计 | `#E5E7EB`（灰白） | 综合汇总 |

### 3.4 灵动岛窗口

#### 3.4.1 三种形态规格

**Compact 形态（空闲状态）**

| 属性 | 值 |
|------|----|
| 宽度 | 动态：等于刘海宽度（约 120-130pt） |
| 高度 | 动态：等于刘海高度（约 34-37pt） |
| 内容 | 仅像素宠物，居中显示 |
| 背景 | 纯黑，圆角 10pt |
| 触发 | `idle` / `error` 状态 |

**Pill 形态（AI 工作中）**

| 属性 | 值 |
|------|----|
| 宽度 | 360pt（固定） |
| 高度 | 34pt（与刘海等高） |
| 内容 | 宠物 ｜ Token 四组数字 |
| 背景 | 纯黑，圆角 10pt |
| 触发 | `thinking` / `streaming` / `awaiting_input` 状态 |
| 展开方向 | 以刘海为中心向两侧等比展开 |

**Expanded 形态（Hover 展开）**

| 属性 | 值 |
|------|----|
| 宽度 | 360pt |
| 高度 | 280pt |
| 内容 | 宠物 + 状态标题 + 四格 Token 卡片 + 历史折线图 |
| 背景 | 黑色（透明度 95%），圆角 20pt |
| 展开方向 | 向下展开，顶部贴刘海底部 |
| 触发 | 鼠标悬停在灵动岛上任意形态 |

#### 3.4.2 形态切换动画

| 切换 | 动画 | 时长 |
|------|------|------|
| Compact → Pill | 水平 spring 展开 | 300ms |
| Pill → Compact | 水平 spring 收缩 | 300ms |
| 任意 → Expanded | 垂直 spring 向下展开 + opacity | 300ms |
| Expanded → 前态 | 垂直 spring 向上收缩 + opacity | 250ms |
| 状态内容切换 | opacity crossfade | 200ms |

#### 3.4.3 窗口技术要求

- `NSPanel`，`borderless`，`nonactivatingPanel`
- WindowLevel = `statusBar + 1`（高于全屏 App 的菜单栏）
- `collectionBehavior = [.canJoinAllSpaces, .stationary, .ignoresCycle, .fullScreenAuxiliary]`
- 背景完全透明（`backgroundColor = .clear`，`isOpaque = false`，`hasShadow = false`）
- 不抢夺键盘焦点
- 屏幕参数变化时自动重新定位

#### 3.4.4 刘海兼容性

| 设备类型 | 处理方式 |
|----------|----------|
| MacBook Pro 2021+ (有刘海) | 精准定位到刘海区域，动态计算坐标 |
| MacBook Air M2+ (有刘海) | 同上 |
| 无刘海 Mac | 降级为屏幕顶部居中浮动胶囊，行为不变 |
| 外接显示器 | 灵动岛显示在内置屏幕，不跟随外接显示器 |

### 3.5 像素宠物

#### 3.5.1 宠物状态与动画

| 状态 | 动画 | 帧数 | 帧率 | 循环 |
|------|------|------|------|------|
| Idle | 闭眼呼吸 | 8 | 8fps | 无限 |
| Idle（随机） | 伸懒腰 | 12 | 8fps | 1次，随机间隔 30-90s |
| Thinking | 眼睛转动 + 冒汗珠 | 8 | 8fps | 无限 |
| Streaming | 嘴巴开合 + 小步移动 | 8 | 12fps | 无限 |
| AwaitingInput | 原地跳脚 + 单手上举 | 10 | 10fps | 无限 |
| Error | 倒地 + 星星特效 | 10 | 8fps | 1次 |
| 进食（叠加） | 张嘴吞 T 字符 + 满足表情 | 8 | 8fps | 1次 |

#### 3.5.2 宠物素材规格

| 属性 | 规格 |
|------|------|
| 单帧尺寸 | 24×24px（@1x），48×48px（@2x） |
| 调色板 | ≤ 16 色 |
| 文件格式 | PNG，透明背景 |
| 命名规范 | `pet_{animation_name}_{frame:02d}.png` |
| 渲染设置 | 图片插值 `.none`（禁止抗锯齿，保持像素感） |

#### 3.5.3 Token 进食系统

- 每累计 1000 个 `output_tokens`，触发一次进食动画
- 进食动画叠加在当前状态动画上方播放（不打断主动画）
- 进食动画播放一次后自动结束，恢复当前状态动画

#### 3.5.4 开发阶段占位方案

正式像素图完成前，使用 SwiftUI Canvas 绘制简化占位宠物：
- 基础形状：椭圆身体 + 圆形头部
- 不同状态用不同颜色区分（蓝/黄/绿/红）
- 眨眼、跳动等简单动画用 SwiftUI `withAnimation` 实现
- 像素图完成后整体替换，接口不变

### 3.6 "等待用户回复"专项功能

这是阶段一中优先级最高、体验要求最严格的功能。

#### 3.6.1 视觉提示（Pill 形态）

- 左侧宠物切换为跳脚动画（焦虑状态）
- 宠物头顶叠加红色脉冲扩散光圈（无限循环）
- 整个 Pill 外边框显示红色脉冲呼吸效果（从透明到 80% 不透明，1.2秒周期）
- 文字区域显示「等待您的回复」（红色 `#EF4444`，加粗）
- 副文字显示等待原因（工具名称 / 命令预览，最多 30 字符，超出截断）
- 右侧显示「回复」按钮（红色背景，点击激活 Codex 窗口）

#### 3.6.2 系统通知

- 状态进入 `awaiting_input` 时，发送一条系统通知
- 通知标题：「Codex 正在等待您的回复」
- 通知内容：工具名称 + 命令预览（工具审批）或问题文本（提问）
- 通知带默认声音
- **防刷屏**：同一会话连续等待时，30 秒内只发一次通知
- 用户未授权通知时，仅保留视觉提示，不报错

#### 3.6.3 等待原因展示规则

**工具审批**（`tool_approval` 类型）：
- 副标题格式：`{工具名}: {命令前30字符}…`
- 示例：`shell: rm -rf /tmp/build && mkdi…`

**提问**（`question` 类型）：
- 副标题格式：Codex 问题文本前 40 字符
- 示例：`请问您希望使用哪个数据库？`

#### 3.6.4 Expanded 面板中的等待详情

Hover 展开后，在 Token 数据上方插入专属区块：
- 红色警示图标 + 「Codex 需要您的确认」
- 工具名称（工具审批）或完整问题文本（提问）
- 命令完整内容（最多 4 行，超出截断）
- 醒目的「前往 Codex 回复」全宽按钮（红色）

#### 3.6.5 一键唤起 Codex

点击「回复」/ 「前往 Codex 回复」按钮时：
1. 在运行中的应用列表查找 Codex（通过 Bundle ID 或应用名匹配）
2. 找到则调用 `activate(options: .activateIgnoringOtherApps)`
3. 未找到则尝试 `NSWorkspace.shared.launchApplication("Codex")`

### 3.7 Hover 展开面板

**触发**：鼠标悬停在灵动岛上任意位置 0ms 后立即开始展开动画

**面板内容（从上到下）**：

1. **状态头部**（高 50pt）
   - 左：当前状态宠物动画（32px）
   - 右：状态标题（中文）+ 副标题（总 Token 用量汇总）

2. **四格 Token 卡片**（高 80pt）
   - 输入 Tokens / 缓存命中 / 实际消耗 / 输出 Tokens
   - 每格：字段标签 + 格式化数字 + 缓存格额外显示命中率

3. **历史折线图**（高 80pt，会话轮数 ≥ 2 时显示）
   - X 轴：最近 20 轮对话轮次
   - Y 轴：每轮 output token 增量
   - 折线颜色：橙色（`#FB923C`）

4. **等待详情区**（仅 `awaiting_input` 状态，高度自适应）

**面板关闭**：鼠标离开灵动岛区域后立即开始收起动画。

## 四、非功能需求

### 4.1 性能要求

| 指标 | 要求 | 测量条件 |
|------|------|----------|
| 内存占用（空闲） | ≤ 35 MB | Activity Monitor，无活跃 Codex 会话 |
| 内存占用（活跃） | ≤ 55 MB | Codex 正在进行长对话 |
| CPU 占用（空闲） | ≤ 0.5% | 无 Codex 活动，宠物 idle 动画运行中 |
| CPU 占用（Streaming） | ≤ 3% | Token 数字每秒更新多次 |
| Token 更新延迟 | ≤ 300ms | 从 JSONL 写入到 UI 数字变化 |
| 状态切换延迟 | ≤ 500ms | 从 Codex 状态变化到灵动岛切换 |
| App 启动时间 | ≤ 1.5s | 冷启动到灵动岛可见 |
| Rust sidecar 内存 | ≤ 15 MB | 单独测量 codex-watcher 进程 |

### 4.2 稳定性要求

- **MTBF**（平均故障间隔）：8 小时以上的连续使用不崩溃
- **Rust sidecar 崩溃恢复**：检测到崩溃后 2 秒内自动重启，App 不受影响
- **JSONL 文件格式变化容错**：Codex 升级后格式小幅变化时，尽量降级处理而非崩溃
- **网络/文件系统异常**：WebSocket 断连、文件被锁定等情况下，静默恢复

### 4.3 兼容性要求

| 维度 | 要求 |
|------|------|
| macOS 最低版本 | macOS Ventura 13.0（`auxiliaryTopLeftArea` API） |
| 架构 | Apple Silicon 原生（arm64）；Intel 按需支持（Universal Binary） |
| Codex 版本 | Codex CLI v0.100+（JSONL sessions 目录格式已稳定） |
| 刘海 Mac 型号 | MacBook Pro 2021 M1 Pro/Max、M2/M3/M4 全系，MacBook Air M2/M3 |

### 4.4 隐私与安全要求

- **完全本地化**：不上传任何用户数据，不访问任何外部服务器
- **最小权限**：仅需要读取 `~/.codex/sessions/` 目录，不需要其他文件系统权限
- **不触碰 JSONL 内容本身**：仅解析 `token_count` 和状态相关字段，不读取 AI 回复的具体文字内容（阶段一）
- **不修改任何 Codex 文件**：阶段一为只读监听，不写入任何 Codex 相关文件
- **App Sandbox 关闭**：明确告知用户原因（需读取 ~/.codex），不在 App Store 分发

### 4.5 分发与更新

- 分发方式：GitHub Releases 提供 `.dmg` 下载
- 安装方式：拖拽 App 到 Applications 文件夹
- 更新机制：集成 Sparkle 框架，支持后台静默检查更新
- 代码签名：Developer ID 签名（非 App Store）+ Notarization

## 五、用户界面规范

### 5.1 视觉风格

- **整体基调**：深色（Dark Mode 优先），与刘海黑色背景融合
- **字体**：SF Mono（数字）/ SF Pro Rounded（状态文字）/ System（设置页）
- **字号层级**：状态标题 13pt / Token 数字 12pt / 标签 9pt / 描述 10pt
- **间距**：基础单位 4pt，组件内 padding 为 8pt
- **圆角**：灵动岛主体 10pt（Compact/Pill）/ 20pt（Expanded）/ 卡片 8pt

### 5.2 动画原则

- 所有动画使用 **spring 曲线**（response ≤ 0.35, dampingFraction ≥ 0.75）
- 避免 linear 动画（显得生硬）
- 宠物动画帧率与状态挂钩，不得使用固定 60fps（能耗浪费）
- Token 数字动画不超过 400ms（响应要快）
- 窗口展开收缩动画不超过 300ms（不拖沓）

### 5.3 文案规范

| 场景 | 文案 | 备注 |
|------|------|------|
| Thinking 状态 | `Thinking…` | 英文，配三点跳动 |
| Streaming 状态 | 不显示文字，只显示 Token 数字 | 数字即信息 |
| AwaitingInput | `等待您的回复` | 中文，红色加粗 |
| 展开面板标题（Idle）| `闲置中` | — |
| 展开面板标题（Thinking）| `正在思考...` | — |
| 展开面板标题（Streaming）| `正在输出` | — |
| 展开面板标题（Awaiting）| `等待您的回复` | — |
| 系统通知标题 | `Codex 正在等待您的回复` | — |
| 一键唤起按钮 | `回复`（Pill） / `前往 Codex 回复`（Expanded）| — |

## 六、边界情况处理

| 情况 | 处理方式 |
|------|----------|
| Codex 未安装 | 设置页显示提示，功能区显示"未检测到 Codex" |
| Codex 未运行 | 灵动岛保持 Idle，不报错 |
| `~/.codex/sessions/` 目录为空 | 等待目录出现，静默等待 |
| 多个 Codex 会话同时运行 | 监听最后修改时间最新的文件（最活跃的会话）|
| Token 值回退（新会话覆盖） | 清零差分基准，重新开始计数 |
| 刘海坐标计算失败 | 降级到屏幕顶部中央 |
| WebSocket 持续无法连接 | 静默降级，仅用 JSONL，不向用户报错 |
| 系统通知权限被拒绝 | 仅视觉提示，不提示用户再次授权（避免烦扰）|
| 宠物图片资源缺失 | 自动使用 SwiftUI Canvas 占位宠物 |
| sidecar 进程持续崩溃 (>5次/分钟) | 停止重试，菜单栏图标变灰，显示错误状态 |

## 七、验收标准

### 7.1 阶段一功能验收

以下所有测试场景通过，视为阶段一完成：

**T01 — 标准对话流**
- 在 Codex 发送问题后 500ms 内，灵动岛从 Compact 切换到 Pill（Thinking 状态）
- AI 开始输出后，Token 数字开始实时递增
- AI 回复完成后 4 秒内，灵动岛收缩回 Compact（Idle 状态）

**T02 — Token 数据准确性**
- StreamingView 展示的累计 Token 总数与 Codex 界面显示误差 < 5%
- 缓存命中率数字随对话轮数合理变化（通常逐渐升高）
- 数字格式化：1234 显示为 `1.2K`，12345 显示为 `12K`

**T03 — 等待状态提示**
- Codex 请求工具审批时，500ms 内灵动岛切换到 AwaitingInput 状态
- 边框出现红色脉冲效果
- 系统通知发送（有权限时）
- 点击「回复」按钮，Codex 窗口被激活并置于前台

**T04 — Hover 展开**
- 鼠标悬停后，面板向下展开，动画流畅
- 四格 Token 卡片数据正确
- 历史折线图在 ≥ 2 轮后显示
- 鼠标离开后面板收起

**T05 — 稳定性**
- 连续使用 2 小时，内存 ≤ 55 MB，无崩溃
- 强制杀死 Codex 进程，灵动岛在 4 秒内回到 Idle，不崩溃
- 强制杀死 sidecar 进程，2 秒内自动重启，恢复正常

**T06 — 无刘海兼容**
- 在无刘海 Mac 上运行，显示顶部浮动胶囊，功能正常
